As far as setting up goes

I'm assuming Git is already downloaded and usable from the command line.
Let's work on some setting configurations. Git needs to know your name right? Open up that command line.
git config --global user.name "My Name"
git config --global user.email myemail@provider.com

Navigate to the repository you want version control on -- Git, let's git init a repo locally. So now we've made
your local repo a Git repo. Meanwhile, find your friend GitHub and make a repository with the same name on there.
If your repository is empty,

  • Simply create a file and add to your repo (like a README). We'll cover commit & pushing later.
Otherwise,
  • Add a remote using
    git remote add origin https://github.com/yourname/yourreponame.git
    git push -u origin master

You can check the status of the modifications you've made to your repo using
git status
You sure you like that? Sweet, let's...

Get ready to commit

Well what exactly is this elusive "commit" thing everyone talks about?
You can think of it as saving something formally -- yeah, you've saved your files, but this time you approve of the changes and
would like to give it a little ID and timestamp.
Which means down the road when you do something catastrophic to your program or project, you can find the last working commit and revert your code!

So commit early and often. Use your best judgement as to what "often" means.

Oh wait. You need the command. Right.
git commit -m "Your commit message here"
Keep that line short and sweet (and descriptive) so you can easily look through your commit history to find things you changed or added.

Then push!

Ok, so you've committed your code, but GitHub doesn't know that because you've done that all locally.
It's not a good idea to keep secrets between Git and GitHub, they're friends after all.
Let GitHub know you've committed changes by
git add filenamehere
git push
At this point, Git may prompt you for your username and password. To speed up this process later on, you can set keys specific to you computer
so you aren't prompted each time.

To experiment with tentative changes,

You can create a git branch using
git checkout -b mybranchname
This will automatically switch you your newest branch. Think of it as your sandbox -- any changes you make here
will not be reflected on your master branch. Break things, write infinite loops, improve your code, whatever.

When you're ready to merge your branched changes to your master branch,
git branch # checks which branch you're on
git merge mybranchname # merges mybranchname onto master branch

And there you have it! Basic branching. If you want to switch between branches, simply git checkout branchname
